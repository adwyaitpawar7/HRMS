PROJECT NAME:
Hotel + Restaurant Management System

PROJECT TYPE:
Enterprise-level Microservices-based Restaurant and Hotel Management Platform.

PRIMARY PURPOSE:
The system is designed to manage restaurant operations, kitchen workflow, inventory management, billing, reporting, authentication, and analytics using scalable microservices architecture.

TECH STACK:
Frontend:
- ReactJS

Backend:
- Java Spring Boot Microservices
- Spring Cloud
- Spring Data JPA
- Hibernate
- Spring Security
- JWT Authentication

Database:
- Oracle Database

Infrastructure:
- Eureka Service Registry
- Spring Cloud Gateway
- Kafka or RabbitMQ for event-driven communication

Build Tool:
- Maven Multi-Module Project

OPTIONAL:
- Docker
- Kubernetes

====================================================
CORE BUSINESS REQUIREMENTS
====================================================

1. RESTAURANT FLOW
- Waiter can create dine-in orders.
- Online customer orders are also supported.
- Orders should appear:
  - in Kitchen Dashboard
  - in Counter/Billing Dashboard

2. KITCHEN WORKFLOW
- Kitchen receives order in real-time.
- Cook can mark:
  - NEW
  - PREPARING
  - READY
- Waiter receives notification when order is READY.
- Counter also receives updates for billing.

3. ORDER TIME TRACKING
System tracks:
- placed_time
- ready_time
- served_time

4. INVENTORY MANAGEMENT
Kitchen inventory management system required:
- Daily stock entry
- Stock usage tracking
- Remaining stock calculation
- Kitchen expense tracking

5. MAKER-CHECKER IMPLEMENTATION
Maker-checker pattern is ONLY required for:
- stock entries
- kitchen expenses

Meaning:
- Maker enters data
- Checker approves/rejects

Fields used:
- mk_user_id
- mk_time
- ck_user_id
- ck_time
- approval_status

Maker-checker should NOT be used in:
- order service
- billing service
- kitchen service

6. REPORTS
Generate:
- daily reports
- monthly reports
- yearly reports

Reports include:
- revenue
- expenses
- profit
- most sold items
- order analytics

====================================================
ARCHITECTURE PRINCIPLES
====================================================

1. MICROSERVICES
Each service:
- owns its own database schema
- has independent business logic
- communicates via REST/events

2. EVENT-DRIVEN COMMUNICATION
System uses:
- Kafka or RabbitMQ

Events include:
- ORDER_CREATED
- ORDER_PREPARING
- ORDER_READY
- PAYMENT_COMPLETED
- STOCK_UPDATED
- EXPENSE_APPROVED

3. SAGA PATTERN
Use choreography-based saga pattern.

Reason:
- maintain distributed consistency
- avoid distributed transactions

Compensation example:
If inventory deduction fails:
- cancel order
- remove kitchen order
- cancel draft bill

4. SOFT DELETE
Important entities should use:
- is_deleted
or
- status-based soft delete

Avoid physical deletion.

5. AUDIT LOGGING
All major tables should include:
- created_by
- created_at
- updated_by
- updated_at

6. COMMON MODULE
common-module stores:
- DTOs
- shared events
- constants
- utility classes
- exception models

- Kafka OR RabbitMQ
(Both supported conceptually)

PURPOSE:
- decouple services
- improve scalability
- improve fault tolerance
- avoid tight coupling
- support asynchronous communication

IMPORTANT:
Services should communicate primarily through events.

REST APIs should mainly be used for:
- synchronous frontend requests
- service-specific queries
- authentication

Backend Services:

- Billing Service
- Report Service

Shared Module:
- common-module

====================================================
EVENT COMMUNICATION PRINCIPLE
====================================================

Instead of:
Order Service directly calling every service,

Use:
Order Service publishes events,
Other services subscribe to them.

Example:

BAD:
Order Service → Kitchen Service
Order Service → Inventory Service
Order Service → Billing Service

GOOD:
Order Service publishes ORDER_CREATED event.
Other services react independently.

====================================================
MAIN DOMAIN EVENTS
====================================================

1. ORDER_CREATED
2. ORDER_PREPARING
3. ORDER_READY
4. ORDER_SERVED
5. PAYMENT_COMPLETED
6. STOCK_UPDATED
7. EXPENSE_APPROVED
8. ORDER_CANCELLED

====================================================
ORDER LIFECYCLE - COMPLETE FLOW
====================================================

----------------------------------------------------
STEP 1 — WAITER CREATES ORDER
----------------------------------------------------

Flow:
Frontend → API Gateway → Order Service

Order Service Actions:
- validate request
- create order
- create order_items
- create order_status_history
- set status = PLACED
- set placed_time

Then publish:

EVENT:
ORDER_CREATED

====================================================
STEP 2 — KITCHEN RECEIVES ORDER
====================================================

Kitchen Service subscribes to:
ORDER_CREATED

Kitchen Service Actions:
- create kitchen_order
- create kitchen_order_items
- set status = NEW
- set received_time

Kitchen Display System shows:
- new order
- table number
- items

====================================================
STEP 3 — INVENTORY RECEIVES EVENT
====================================================

Inventory Service subscribes to:
ORDER_CREATED

Inventory Service Actions:
- calculate ingredient usage
- deduct ingredients
- create stock_usage
- create stock_transaction_history

Example:
Biryani requires:
- rice
- oil
- vegetables

System deducts quantities automatically.

====================================================
STEP 4 — COOK STARTS PREPARATION
====================================================

Cook clicks START in Kitchen Dashboard.

Kitchen Service Actions:
- update kitchen_order status = PREPARING
- set start_time

Publish:
ORDER_PREPARING

Order Service receives event:
- update order status = PREPARING

====================================================
STEP 5 — ORDER READY
====================================================

Cook clicks READY.

Kitchen Service Actions:
- update kitchen_order status = READY
- set ready_time

Publish:
ORDER_READY

Order Service Actions:
- update order status = READY
- set ready_time

Waiter Dashboard:
- show READY orders

Counter Dashboard:
- show ready-for-billing orders

====================================================
STEP 6 — WAITER SERVES FOOD
====================================================

Waiter clicks SERVED.

Order Service Actions:
- update status = SERVED
- set served_time

Publish:
ORDER_SERVED

====================================================
STEP 7 — BILL GENERATION
====================================================

Counter creates bill.

Billing Service Actions:
- create bills record
- create bill_items
- calculate tax
- calculate total amount

Bill Status:
GENERATED

====================================================
STEP 8 — PAYMENT PROCESSING
====================================================

Customer pays.

Billing Service Actions:
- create payments record
- update bill status = PAID

Payment Methods:
- CASH
- CARD
- UPI
- ONLINE

Publish:
PAYMENT_COMPLETED

====================================================
STEP 9 — REPORT SERVICE UPDATES
====================================================

Report Service subscribes to:
- PAYMENT_COMPLETED
- STOCK_UPDATED
- EXPENSE_APPROVED

Report Service Actions:
- update daily reports
- update monthly reports
- update yearly reports
- update item analytics

====================================================
TABLE SESSION WORKFLOW
====================================================

IMPORTANT DESIGN DECISION:
Use table_sessions.

Purpose:
Prevent multiple disconnected orders for same table.

Flow:
1. Customer sits at table.
2. Create table_session.
3. All order items belong to same session.
4. Bill generated for session.
5. Session closes after payment.

Benefits:
- easier billing
- supports incremental ordering
- tracks table occupancy

====================================================
SAGA PATTERN IMPLEMENTATION
====================================================

STYLE:
Choreography-Based Saga Pattern

PURPOSE:
Maintain consistency across distributed microservices.

Avoid:
Distributed transactions.

====================================================
WHY SAGA IS NEEDED
====================================================

Example Failure Scenario:

1. Order created
2. Kitchen order created
3. Billing draft created
4. Inventory deduction fails

Without saga:
system becomes inconsistent.

====================================================
SAGA COMPENSATION FLOW
====================================================

If inventory deduction fails:

1. Publish INVENTORY_FAILURE
2. Order Service cancels order
3. Kitchen Service removes kitchen order
4. Billing Service cancels draft bill

Compensation Event:
ORDER_CANCELLED
